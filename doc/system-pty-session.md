# Main Process & PTY Management

This system is the backbone of HyperTerm Canvas, running within the Bun process (the Electrobun main thread). It is responsible for spawning, managing, and destroying the underlying shell processes, as well as capturing their standard input/output.

## Core Components

- **`PtyManager`** (`src/bun/pty-manager.ts`): Wraps `Bun.spawn` with the `{ terminal: true }` option, enabling native PTY capabilities without relying on external dependencies like `node-pty`. It manages the standard IO channels, parses UTF-8 output streams safely, and captures the file descriptors allocated for the Sideband protocol (fd3, fd4, fd5).
- **`SessionManager`** (`src/bun/session-manager.ts`): A higher-level orchestra that manages multiple `PtyManager` instances ("surfaces"). It acts as the bridge connecting standard streams and sideband channels to the JSON-RPC layer and the webview. It also maintains a recent history buffer (up to 64KB) of terminal output per surface, useful for CLI commands like `ht read-screen`.

## Flow

1. A new workspace or split is requested via the UI or CLI.
2. `SessionManager` creates a new `PtyManager` instance.
3. The underlying process is started with specialized environment variables (like `TERM=xterm-256color` and specific FD pointers: `HYPERTERM_META_FD=3`).
4. Output generated by the child process is intercepted, buffered, and pushed up to `SessionManager`, which then triggers a callback (`onStdout`) piped to the Electrobun webview.
5. Exit codes and process terminations bubble up, destroying the respective `PtyManager` and removing the surface.

## Critiques & Limitations

- **Signal Handling Constraints:** The `PtyManager.kill(signal)` method attempts to map standard signals (SIGTERM, SIGINT) to numeric values, but Bun's `subprocess.kill()` is relatively basic. Graceful termination of deep process trees (like a running dev server spawned by a shell script) might fail to propagate the signal accurately.
- **Fixed Output Buffer Limit:** `SessionManager` limits output history to 64KB (`MAX_HISTORY_BYTES`). While this prevents memory bloat, it heavily restricts what the user can read using `ht read-screen --scrollback`. In heavily noisy environments (like a long build log), the history will truncate too quickly.
- **No Automatic Restart:** If a shell crashes unexpectedly (e.g., segfault), the surface simply closes without offering the user a chance to inspect the error or respawn the pane (like tmux's `remain-on-exit` capability).
