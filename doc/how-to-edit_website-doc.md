# How to edit `website-doc/`

The τ-mux docs site lives at `website-doc/` and ships to GitHub Pages
under `/TauMux/`. It is an **Astro Starlight** site, fully static, with
its own `package.json` independent of the root project.

This doc covers the editing rules that aren't obvious from the README:
when to bump the app version, how the French translation mirror works,
and the small staging conventions that keep the two locales in sync.

The site README (`website-doc/README.md`) has the basics — folder →
sidebar mapping, frontmatter cheat-sheet, `bun run dev`. Read that
first if you've never touched the site before.

## TL;DR

- Edit Markdown under `website-doc/src/content/docs/`. Sidebar is
  autogenerated from the folder layout.
- **English source is the truth.** Whenever you edit an English page,
  edit the matching `fr/` page in the same commit.
- If your change visibly affects the app (new RPC, new CLI verb, new
  setting, new keyboard shortcut, fixed bug surfaced to users) bump
  the app version with `bun run bump:patch` *before* committing.
- Stage notes for human-driven sweeps in `doc/changes_to_document.md`.

## Local dev

```bash
cd website-doc
bun install                       # first time only
bun run dev                       # http://localhost:4321 — live reload
bun run build                     # static output in dist/
bun run preview                   # serves dist/ locally
```

`bun run build` is the canonical "did I break anything" check. It runs
all locales together, builds Pagefind, and exits non-zero on any link
or schema error.

## Adding or editing a page

1. **English first.** Drop a `.md` (or `.mdx`) into the right folder
   under `src/content/docs/`. The folder picks the sidebar group.
2. Frontmatter:
   ```yaml
   ---
   title: My new page
   description: One-line summary used in nav and meta tags.
   sidebar:
     order: 3                     # optional, lower = earlier
   ---
   ```
3. Write the body. Internal links use root-relative paths
   (`/concepts/architecture/`) — `astro.config.mjs` has a remark
   plugin that prefixes the `/TauMux/` base automatically.
4. **Mirror the page in French.** Write `src/content/docs/fr/<same-path>`
   with the same frontmatter (translated `title` / `description`),
   same headings, and the body translated. See "French translation"
   below.
5. Run `bun run build` to confirm both locales render.

**Adding a brand-new top-level section** requires one edit to
`astro.config.mjs` — append to the `sidebar` array with a `label` plus
a `translations: { fr: "…" }` entry — then create the matching folders
under both `docs/` and `docs/fr/`.

## App version

The app version (`package.json` → `version`) is hard-coded in a few
places that ship to users. The site's `<Default>` site-title pill reads
it directly from the root `package.json` at build time, but two doc
pages have it embedded in example output and need explicit bumps.

### When to bump

Bump before any commit that **visibly changes app behavior**:
new RPC method, new CLI verb, new setting, new keyboard shortcut,
behavior change a user could observe. Pick the level:

```bash
bun run bump:patch                # bug fixes, polish, docs that follow code
bun run bump:minor                # new feature
bun run bump:major                # breaking change
```

The script (`scripts/bump-version.ts`) updates **all** of:

- `package.json` — npm version
- `electrobun.config.ts` — bundle `CFBundleVersion`
- `src/bun/rpc-handlers/system.ts` — what `system.version` returns
- `website-doc/src/content/docs/cli/system.md` — example `ht version` output
- `website-doc/src/content/docs/api/system.md` — example JSON-RPC payload
- `website-doc/src/content/docs/fr/cli/system.md` — French mirror
- `website-doc/src/content/docs/fr/api/system.md` — French mirror

Review the diff, then commit.

### When NOT to bump

- Pure docs work that doesn't follow a code change (typos, rewrites,
  new translation, restructuring).
- Internal refactors with no user-visible effect.
- Test-only changes.
- Tracking files under `doc/`.

If you skip the bump, **say so in the commit message** — one line is
enough ("Docs-only — no app code touched, so skipping the version
bump.") so reviewers don't assume you forgot.

## French translation

The site is bilingual. English at the root, French at `/fr/`,
configured in `astro.config.mjs` via Starlight's i18n:

```js
defaultLocale: "root",
locales: {
  root: { label: "English", lang: "en" },
  fr:   { label: "Français", lang: "fr" },
},
```

Starlight renders a language picker in the header. A custom
`Head.astro` injects a one-shot synchronous redirect that matches
`navigator.languages` against the supported locales on first visit and
persists the choice in `localStorage` so it never overrides a manual
click on the picker.

### Mirror rule

**Every English page has exactly one French page at the same path under
`fr/`.** When you edit an English page, edit the French one in the same
commit. When you add a new English page, add the French one. When you
delete one, delete both. CI doesn't enforce this — discipline is on the
author. `find src/content/docs/fr -type f | wc -l` should match
`find src/content/docs -maxdepth 4 -type f -not -path "*/fr/*" | wc -l`.

### What to translate, what to keep in English

Translate:

- Prose, headings, list items, table description columns.
- Frontmatter `title` and `description`.
- Image `alt` text, link labels.

Keep in English:

- All code blocks and inline code.
- Command names, flags, file paths, env var names.
- Identifiers, function names, RPC method names, type names, settings keys.
- Framework / product names (Electrobun, xterm.js, Bun, Astro,
  Starlight, WebKit, JSON-RPC, WebSocket, PTY, fd 3/4/5, …).
- Acronyms.
- Keyboard shortcuts (`⌘⇧P`).

### Internal links

Internal links inside French pages must point to French targets:

```md
<!-- in src/content/docs/fr/concepts/architecture.md -->
See [the PTY model](/fr/concepts/pty-model/).
```

The remark plugin in `astro.config.mjs` adds the `/TauMux/` base prefix
automatically — you write `/fr/concepts/...`, the published HTML has
`/TauMux/fr/concepts/...`. Anchors (`#section`) and external `https://`
links are left alone.

## Staging notes for the next sweep

`doc/changes_to_document.md` is a running scratchpad of changes that
landed in the app but haven't yet been folded into `website-doc/`.
Treat it as a queue:

1. When you ship a feature without updating `website-doc/` in the same
   PR, add an entry under the new version's section. Each entry should
   capture *what changed* and *which docs page should reflect it*.
2. When you do a docs sweep, work through the entries from oldest to
   newest, then **delete them from the file** — that's how the queue
   stays bounded.

Per CLAUDE.md, the project rule is to keep this file up to date and
clear it whenever you do a docs sweep against the live site.

## Verifying a docs change

```bash
cd website-doc
bun run build                     # builds both locales, fails on broken links
```

Successful output: `[build] N page(s) built in …` where N is roughly
twice the number of source files (EN + FR + index pages). Any
404-on-internal-link or schema validation error blocks the build.

For visible polish work, `bun run dev` and click through the changed
pages in both `/` and `/fr/` — the language picker is in the header.
