# crazyShell Reviewer Report

## Executive Summary

This change adds a substantial visual-review pipeline, but it currently behaves more like a manual gallery generator than a production-grade regression gate. The main risks are not in PTY correctness itself; they are in the surrounding verification surface:

- the new design report does not fail CI when regressions are detected,
- the report does not detect baseline-only/deleted shots,
- one committed baseline is already missing for a covered native state,
- and the new 700+ LOC report/promotion tooling has no direct tests.

The harness direction is strong and mostly aligned with the repo docs: the web mirror now mirrors sideband traffic in `tests-e2e/server-boot.ts`, and the shell selection change in `SessionManager` is consistent with the PTY/session guidance. The weak point is the quality-gate story around the new review artifacts.

## Highest Value Findings

### 1) Visual regressions are reported, but not enforced anywhere in the build pipeline

- Severity: high
- Confidence: high

#### Why it matters

The repository’s own standards call for strong quality gates (`DEV_RULES.md:99-147`) and the external architecture guide explicitly pushes “quality gate” muscle memory. This change adds a visually rich report, but not an actual gate. That creates a false sense of protection: the suite can produce red cards, missing shots, or broken demo states while `test:full-suite` still exits successfully.

For a terminal product where overlays must not silently drift away from PTY reality, an advisory-only report is too weak as the primary verification surface.

#### Evidence

- `package.json:30-36`
  - `report:design` and `test:full-suite` both end by running `bun scripts/build-design-report.ts`.
  - There is no follow-up command that fails on `over`, `new`, `missing`, or `dim-mismatch`.
- `scripts/build-design-report.ts:149-308`
  - The script counts `overCount`, `newCount`, and `missingCount`, writes HTML/JSON, and only logs:
    - `... ${shots.length} shot(s), ${overCount} over threshold, ${newCount} new, ${missingCount} missing`
  - It never exits non-zero for regression states.
- `tests-e2e/design/demos.spec.ts:10-11`
  - The suite explicitly says: “Tests never fail on a demo’s internal error; they still snap...”
- `doc/design-report.md:36-42`
  - The workflow is explicitly human-review-first: red cards in the report, then manual promotion. That is useful, but it is not a CI gate.

#### Proposition

Add an explicit gated mode for CI, rather than changing local ergonomics:

- Keep the current report builder for human review.
- Add a `--fail-on-regression` mode or separate script that exits non-zero when:
  - any shot is `over`,
  - any shot is `missing`,
  - any shot is `dim-mismatch`,
  - and optionally any unexpected `new` shot appears in gated suites.
- Wire CI-facing scripts to the gated mode:
  - e.g. `test:full-suite` or a dedicated `test:design:gate`.
- For demo suites, add at least a minimal correctness assertion before snapshotting:
  - a known sideband panel selector exists,
  - or expected terminal marker text appears,
  - or a known content block count is non-zero.

That preserves the manual gallery while restoring an actual quality gate.

#### Tests to add or update

- Unit test for regression-gate exit behavior:
  - synthetic report data with one `over` shot must return non-zero in gated mode.
- Unit test for `missing` and `dim-mismatch` exit behavior.
- One e2e smoke test that intentionally produces a known red shot and verifies the gate fails.
- Demo-spec assertions:
  - at least one sideband/demo presence assertion before `snap()`.

#### Docs/skills/workflows to update

- `doc/design-report.md`:
  - distinguish “manual review mode” vs “CI gate mode”.
- `DEV_RULES.md` / repo workflow docs:
  - mention the visual regression gate explicitly if adopted.
- Any reviewer/verification workflow docs under `doc/` that currently imply the report itself is sufficient.

### 2) The report cannot detect baseline-only/deleted screenshots, so coverage shrinkage is invisible

- Severity: high
- Confidence: high

#### Why it matters

This is the most important correctness gap in the report builder itself.

If a screenshot disappears from the current run because:
- a test was removed,
- a shot name changed,
- a scenario silently stopped being emitted,
- or a fixture regressed and no longer produces a capture,

the report currently has no way to surface that as a missing-from-current regression. That means review coverage can shrink without producing a visible red card.

For a visual baseline system, “missing current shot” is not a cosmetic issue; it is a regression in the verification contract.

#### Evidence

- `scripts/build-design-report.ts:64-85`
  - `ReportShotFixed.status` includes `"baseline-only"`.
- `scripts/build-design-report.ts:155-280`
  - The main loop iterates only over current index entries:
    - `for (const e of entries) { ... }`
  - It checks whether a matching baseline exists for each current shot.
  - It never scans `tests-e2e-baselines/` for files that are absent from the current run.
- `scripts/build-design-report.ts:290-307`
  - Summary includes `overCount`, `newCount`, `missingCount`.
  - There is no `baselineOnlyCount`.
- Prior art already exists:
  - `scripts/diff-screenshots.ts:203-235` explicitly computes:
    - `missingInCurrent` (baseline-only)
    - `missingInBaseline` (current-only)

So the repo already had a stronger diff concept; the new report regressed that property.

#### Proposition

Make baseline coverage symmetric:

- Enumerate `tests-e2e-baselines/<suite>/*.png`.
- Build the same canonical key used for current entries.
- For any baseline without a current shot, emit a synthetic report row with:
  - `status: "baseline-only"`
  - `shotRel: ""`
  - `baselineRel` populated
  - metadata derived from filename where possible.
- Add `baselineOnlyCount` to the summary and the UI filter.
- In gated mode, treat `baseline-only` as a failure unless explicitly allowed.

This preserves the contract-first idea of “the expected review surface is known and typed”.

#### Tests to add or update

- Unit test: baseline file present, no current entry => row with `status: baseline-only`.
- Unit test: summary includes `baselineOnlyCount`.
- Unit test: HTML/data.json includes baseline-only entries and filters them correctly.
- Regression test against a tiny synthetic baseline/current fixture set.

#### Docs/skills/workflows to update

- `doc/design-report.md`:
  - add “baseline-only / missing current” to the report semantics.
- `tests-e2e-baselines/README.md`:
  - document that deleted/renamed shots are surfaced, not silently ignored.

### 3) The native suite captures `rename-workspace`, but the committed baseline set is already incomplete

- Severity: medium
- Confidence: high

#### Why it matters

A baseline system loses signal quickly if it permanently reports “new” artifacts that are not actually new. Reviewers learn to ignore the noise. This is already happening in the very first baseline drop.

One of the documented native states is exercised in the spec, but the corresponding baseline PNG is missing from `tests-e2e-baselines/native/`. That means every full run will keep showing at least one “new” native shot unless someone manually notices and promotes it.

#### Evidence

- `tests-e2e-native/specs/design-review.spec.ts:165-178`
  - test: `component-rename-workspace-dialog`
  - snapshot step: `dialog-rename-workspace`
- `doc/design-report.md:102-104`
  - explicitly says native Tier-2 states include:
    - `rename-workspace + rename-surface dialogs`
- Baseline inventory search under `tests-e2e-baselines/native/`
  - present: `design-review-component-rename-surface-dialog-dialog-rename-surface.png`
  - absent: any `rename-workspace` PNG
- `tests-e2e-baselines/README.md:13-18`
  - describes native baseline naming/usage, so this omission directly affects the documented workflow.

I also compared expected native shot names against committed native baselines; `dialog-rename-workspace` is the missing one.

#### Proposition

Make the baseline set internally consistent before relying on it:

- Either commit the missing baseline:
  - expected filename shape: `design-review-component-rename-workspace-dialog-dialog-rename-workspace.png`
- Or, if that state is not stable enough yet, explicitly skip/remove that snapshot from the design-review spec until it is promotable.
- Add a lightweight manifest check:
  - expected design-review shot names vs committed baseline files.
  - This should fail if a covered snapshot is missing from the committed baseline set.

#### Tests to add or update

- Manifest test for native design-review baseline completeness:
  - enumerate expected `app.snap(...)` names from `design-review.spec.ts` and compare to committed baselines.
- Optionally extend the report builder test fixtures to verify that every documented design-review state can resolve to either a baseline or an intentional “new” allowlist.

#### Docs/skills/workflows to update

- `tests-e2e-baselines/README.md`
- `doc/design-report.md`
- Any internal reviewer workflow that assumes “new” always means intentional change.

### 4) The new report/promotion pipeline adds substantial logic with zero direct tests

- Severity: medium
- Confidence: high

#### Why it matters

This PR adds a fairly large amount of non-trivial logic:
- JSONL parsing
- latest-entry reduction
- PNG copying and dimension checks
- pixel diffs
- HTML generation
- baseline promotion
- native/web staging behavior

Yet none of it is directly covered by unit tests. Bugs here will only surface after expensive Playwright runs, and some failures are silently downgraded (`catch { status = "new"; }` style behavior). That is a maintainability risk and a verification risk.

This also drifts from the architecture guide’s preference for explicit contracts and testable focused modules.

#### Evidence

- `scripts/build-design-report.ts` is 715 LOC.
- `scripts/promote-design-baseline.ts` adds promotion semantics with destructive behavior.
- Search under `tests/` for:
  - `build-design-report`
  - `promote-design-baseline`
  - `screenshots-index.jsonl`
  - `design-report`
  returned no direct tests.
- `scripts/build-design-report.ts:252-254`
  - diff/read errors fall back to `status = "new"`, which is especially important to test.
- `scripts/build-design-report.ts:87-118`
  - reduction logic (`latestPerKey`) is pure and easy to unit test.
- `scripts/build-design-report.ts:127-134`
  - annotation parsing / path resolution are pure and easy to unit test.

#### Proposition

Refactor the script into testable pieces and add Bun tests around them:

- Extract pure helpers:
  - `readIndexLines()`
  - `latestPerKey()`
  - `buildShotKey()`
  - `classifyShotAgainstBaseline()`
  - `enumerateBaselineOnly()`
  - `renderReportData()`
- Keep filesystem orchestration thin in the top-level script.
- Add tiny fixture PNGs and synthetic JSONL entries under `tests/fixtures/design-report/`.

This is exactly the kind of tooling that benefits from contract-first, focused-module design.

#### Tests to add or update

- `latestPerKey()` keeps newest entry by timestamp.
- corrupt/missing PNG handling does not silently misclassify regressions.
- dimension mismatch produces `dim-mismatch`.
- current-without-baseline produces `new`.
- baseline-without-current produces `baseline-only`.
- promotion copies only staged shots and handles suite directories intentionally.

#### Docs/skills/workflows to update

- `doc/design-report.md`
  - add a short “how this is verified” section once tests exist.
- Any internal test workflow docs that point maintainers at the design suite.

## Verdict

- Good architectural direction: the sideband wiring in `tests-e2e/server-boot.ts` and the shell handling change in `SessionManager` move the harness closer to production behavior.
- The biggest issue is verification, not implementation: the new visual suite is advisory, not gating.
- The report currently misses an important regression class: baseline-only/deleted shots.
- The committed baseline set is already inconsistent with the exercised native coverage.
- The new 700+ LOC report toolchain needs direct unit coverage before it becomes trusted infrastructure.
- I would not block the general direction of this change, but I would treat the quality-gate gaps as follow-up work that should land before maintainers rely on this suite as a production signal.
