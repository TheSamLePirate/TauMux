# crazyShell Reviewer Report

## Executive Summary

This incremental review focused on the new repository-local review-agent workflow added in `57e6c66`. The change is directionally useful, and the basic helper layer is clean and type-safe, but the execution path is not yet production-trustworthy.

The two biggest issues are safety and attribution:

- the runner claims proposition-only behavior while invoking Hermes in an effectively unrestricted mode;
- the saved report is attributed to `HEAD` even when the working tree is dirty, so a report can silently include uncommitted changes while looking commit-pure.

After that, the main risks are portability and verification drift:

- the workflow is pinned to a machine-specific absolute architecture-guide path;
- the important runner/watcher behavior is largely untested, and the repository `typecheck` does not cover the new `scripts/*.ts` entrypoints at all.

## Highest Value Findings

### 1. The “proposition-only” reviewer is launched with mutation-capable privileges

- Severity: high
- Confidence: high

#### Why it matters

The workflow documentation repeatedly says the reviewer is proposition-only and should not modify code. The implementation does not enforce that contract; it relies on prompt discipline and a best-effort after-the-fact git-status warning.

That is not strong enough for a repo-local automation path. It violates the architecture guide’s contract-first principle and DEV_RULES’ emphasis on explicit boundaries. A code reviewer with shell access in `--yolo` mode is not an inspection-only subsystem.

#### Evidence

- The docs claim proposition-only behavior:
  - `doc/code-review-agent.md:135-138`
  - `code_reviews/README.md:7-10`
  - `code_reviews/README.md:50-52`
  - `README.md:549-558`
  - `doc/SKILLS.md:531-541`
- The runner launches Hermes with broad, side-effect-capable permissions:
  - `scripts/run-code-review-agent.ts:51-66`
  - specifically:
    - `--yolo`
    - tool profile `terminal,file,todo,skills`
- The only enforcement is a post-run status comparison:
  - `scripts/run-code-review-agent.ts:80-91`
  - `scripts/run-code-review-agent.ts:122-129`
  - `scripts/run-code-review-agent.ts:163-169`

That detection is materially weaker than prevention:
- it cannot prevent writes;
- it only inspects git status deltas, not other side effects;
- it still writes and blesses the report even when mutations are detected.

#### Proposition

Make the execution model actually read-only.

Recommended direction:
1. Remove `--yolo` for this workflow.
2. Restrict the tool profile to inspection-only capabilities.
3. If Hermes cannot be sandboxed sufficiently, run the review in a temporary detached worktree or clone and discard it afterward.
4. Treat any detected mutation outside `code_reviews/` as a hard failure, not a warning banner.
5. Document the exact enforcement mechanism so the contract is technical, not aspirational.

#### Tests to add or update

- Add runner-level tests that assert the Hermes invocation flags and tool set.
- Add a failure-path test that simulates a post-run mutation outside `code_reviews/` and verifies the command exits non-zero instead of saving a “warning” report.
- Add a regression test for the markdown sanitizer on normal, fenced, and malformed outputs.

- `doc/code-review-agent.md`
- `code_reviews/README.md`

### 2. Reports can be attributed to a clean commit while actually reviewing a dirty working tree

- Severity: high
- Confidence: high

#### Why it matters

This workflow records the report as reviewing a specific commit and advertises the watcher as triggering on “the current HEAD commit.” But the implementation does not require a clean working tree. That means the report can include local unstaged/uncommitted changes while its filename and frontmatter claim it reviewed only `HEAD`.

That poisons the reliability of the review history. It also weakens the incremental-review baseline, because later runs may treat a report generated from dirty state as a trusted snapshot of the commit.

#### Evidence

- The runner captures current status but does not gate on cleanliness:
  - `scripts/run-code-review-agent.ts:26`
  - `scripts/run-code-review-agent.ts:35-45`
- The report is always stamped with `HEAD`:
  - `scripts/run-code-review-agent.ts:27-30`
  - `scripts/run-code-review-agent.ts:111-117`
- The watcher runs whenever `HEAD` lacks a report, regardless of dirty state:
  - `scripts/run-code-review-watch.ts:17-33`
  - `scripts/run-code-review-watch.ts:47-56`
- The docs describe the output as commit-scoped:
  - `code_reviews/README.md:9-10`
  - `code_reviews/README.md:50`
  - `README.md:551`
  - `doc/SKILLS.md:541`

The prompt includes working-tree status, but that is not a substitute for attribution correctness:
- `src/shared/code-review-agent.ts:97-99`

#### Proposition

Enforce clean-state semantics for commit-attributed reports.

Recommended direction:
1. Refuse to run when `git status --short` is non-empty, unless an explicit `--allow-dirty` flag is passed.
2. If dirty mode is allowed, change the output contract:
   - mark the report as `dirty_worktree: true` in frontmatter;
   - do not imply commit-pure review in the filename or README copy.
3. In watch mode, skip runs while dirty and log a clear reason.

#### Tests to add or update

- Add runner tests for:
  - clean tree → report allowed;
  - dirty tree without override → fail fast;
  - dirty tree with override → report explicitly marked dirty.
- Add watcher tests ensuring it skips dirty working trees and retries later.

#### Docs/skills/workflows to update

- `doc/code-review-agent.md`
- `code_reviews/README.md`
- `README.md`
- `doc/SKILLS.md`

### 3. The workflow is hardwired to a user-specific architecture-guide path

- Severity: medium
- Confidence: high

#### Why it matters

The review feature is now advertised as a normal repository command, but its core prompt contract depends on `/Users/olivierveinand/Documents/DEV/architecture-guide.md`. That is a local-machine convention, not a portable repository dependency.

This creates a fragile workflow surface:
- it is not reproducible on another machine without recreating that exact path;
- it is awkward for CI or other reviewers;
- the docs present it as a general command, but the implementation has a hidden local prerequisite.

This is a maintainability problem and an architecture drift problem: repo behavior depends on undeclared host-local state.

#### Evidence

- Hardcoded absolute path in the runner:
  - `scripts/run-code-review-agent.ts:21-22`
  - reused in report frontmatter:
    - `scripts/run-code-review-agent.ts:116`
- The prompt hardcodes the same absolute path into the review contract:
  - `src/shared/code-review-agent.ts:79`
- The docs repeat the same host-specific path as normative:
  - `doc/code-review-agent.md:97`
- There is no path resolution, existence check, env override, or fallback copy in the repo.

#### Proposition

Make the architecture-guide dependency explicit and configurable.

Recommended direction:
1. Resolve the guide path from an environment variable first, with the current absolute path only as a local fallback.
2. Validate existence before invoking Hermes; fail with a clear error if missing.
3. Better: vendor a repo-local architectural reference or a checked-in excerpt if this workflow is intended for other contributors.
4. Record both the logical source and resolved path in frontmatter, not just an absolute host path.

#### Tests to add or update

- Add tests for guide-path resolution precedence:
  - env override;
  - local default;
  - missing file failure.
- Add tests that the built prompt includes the resolved path actually used.

#### Docs/skills/workflows to update

- `doc/code-review-agent.md`
- `README.md`
- `code_reviews/README.md`

### 4. The verification surface is too shallow: core runner/watcher behavior is untested, and `typecheck` does not cover the new scripts

- Severity: medium
- Confidence: high

#### Why it matters

Most of the important behavior lives in the executable scripts, not in the shared helpers. Right now the test suite only covers string-building helpers, while the live behavior includes git interactions, Hermes invocation, mutation detection, polling semantics, and report writing.

Separately, the repository `typecheck` command does not include `scripts/` or `tests/`. That means the new review-agent entrypoints can regress without being caught by the required “typecheck passed” gate described in `DEV_RULES.md`.

For a workflow tool that writes review artifacts and orchestrates external processes, this is too weak.

#### Evidence

- `tests/code-review-agent.test.ts:1-77` only exercises helper functions from `src/shared/code-review-agent.ts`.
- No test references were found for:
  - `scripts/run-code-review-agent.ts`
  - `scripts/run-code-review-watch.ts`
  - `findLatestReviewedCommit`
  - `diffUnexpectedStatus`
  - `readPollSeconds`
  - `sanitizeMarkdown`
- The required scripts were added in `package.json`:
  - `package.json:26-27`
- But the repository `typecheck` target only includes `src/**/*`:
  - `tsconfig.json:20-21`
- I ran `bun run typecheck` and it passed, which confirms the current gate does not validate these new script files.

This conflicts with the repo’s own verification rules:
- `DEV_RULES.md:101-117`
- `DEV_RULES.md:139-147`

#### Proposition

Expand the quality surface to include the actual executable code.

Recommended direction:
1. Add script-level tests by isolating pure functions from:
   - git state discovery;
   - review-document construction;
   - dirty-tree/mutation handling;
   - watcher poll interval resolution;
   - has-review detection.
2. Split side-effectful shells from pure orchestration so they can be tested without invoking Hermes.
3. Make `typecheck` cover `scripts/` and `tests/`, either by:
   - expanding `tsconfig.json`, or
   - adding a dedicated `tsconfig.scripts.json` / `tsconfig.tests.json` and including them in CI.

#### Tests to add or update

- Unit tests for:
  - `readPollSeconds` precedence and invalid values;
  - `hasReviewForCommit` matching behavior;
  - dirty-tree gating;
  - mutation detection behavior;
  - report frontmatter generation;
  - latest-review selection logic.
- Integration-style tests with mocked `spawnSync`/filesystem for both scripts.

#### Docs/skills/workflows to update

- `DEV_RULES.md` only if the intended verification scope is changed
- otherwise update implementation and CI to match the existing rule set
- `doc/code-review-agent.md` to document the expected automated checks for the workflow
