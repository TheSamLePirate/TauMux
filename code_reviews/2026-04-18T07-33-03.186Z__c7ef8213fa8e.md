# crazyShell Reviewer Report

## Executive Summary

This increment improves the review agent’s posture by moving from “warning report” to “hard fail on detected mutation,” but the implementation still falls short of the repository’s stated proposition-only guarantee.

The biggest issue is architectural, not cosmetic: `review:agent` still launches Hermes with write-capable tooling and `--yolo`, while the new guard only compares git status inside the repo. That means the run is not actually constrained to inspection-only behavior, and some mutation classes remain invisible. The second major issue is verification drift: the production entrypoints are `scripts/*.ts`, but `bun run typecheck` excludes `scripts/`, and the tests only cover pure helpers in `src/shared/`. The safety story therefore relies on untyped, unintegration-tested script code.

I would not call this change “more secure” yet. It is directionally better, but the enforcement boundary is still porous.

## Highest Value Findings

### 1. The reviewer is still not actually sandboxed or inspection-only

- Severity: high
- Confidence: high

#### Why it matters

The docs now claim the review run is “inspection mode” and invalid if it mutates outside `code_reviews/`. The implementation does not enforce that boundary. It gives Hermes write-capable tools and broad autonomy, then only checks repo-local git status after the fact.

That is an architectural mismatch with both `DEV_RULES.md` and the external architecture guide: the safety boundary is declared in docs/prompt text, but not enforced at the execution boundary. In production terms, this is policy-by-prompt, not policy-by-contract.

#### Evidence

- `scripts/run-code-review-agent.ts:53-68` launches:
  - `hermes --yolo chat`
  - tool set `terminal,file,todo,skills`
- `scripts/run-code-review-agent.ts:82-86` only checks `git status --short` before/after the run.
- `README.md:549-558` says the runner is effectively inspection-only and invalidates mutations outside `code_reviews/`.
- `doc/code-review-agent.md:64-72, 133-139` describes the run as proposition-only / inspection-only.
- `src/shared/code-review-agent.ts:93-100` only tells the model not to write files; it does not remove the capability to do so.

Practical consequence: a Hermes run can still write to:
- arbitrary absolute paths reachable by file tools
- `~/.hermes/skills/...` via skill tooling
- any non-git-tracked location outside the repo

None of those are detected by `git status --short` on the repository.

#### Proposition

Move from prompt-based restraint to an enforced read-only boundary.

Specific fix direction:
- Do not expose write-capable tools to the review run. Keep report writing in the runner process only.
- If Hermes CLI cannot expose read-only file tools separately, run Hermes inside an OS-level sandbox / read-only worktree / restricted HOME so writes outside the report output path are impossible.
- Treat `skills` as disallowed for this workflow. A code reviewer should not be able to mutate persistent agent state during a review run.

#### Tests to add or update

- Add an integration test for `scripts/run-code-review-agent.ts` with a stubbed Hermes executable that attempts:
  - a repo-external file write
  - a skill mutation
  - a repo-local write outside `code_reviews/`
- Assert the runner blocks or fails each case, not just the repo-local one.

#### Docs/skills/workflows to update

- `doc/code-review-agent.md`
- `README.md`

- Any repo-local reviewer skill/workflow docs that describe the run as “inspection-only”

### 2. The new hard-fail mutation detector misses changes to already-dirty files

- Severity: high
- Confidence: high

#### Why it matters

The new safeguard only detects newly appearing `git status --short` lines. If a file is already dirty before the review starts, Hermes can mutate it further and the status line remains identical, so the runner reports no unexpected mutation.

That is a direct correctness hole in the safety mechanism. It means the “hard-fail on mutation” promise is false whenever the working tree is not pristine.

#### Evidence

- `scripts/run-code-review-agent.ts:28` captures `beforeStatus = gitLines(["status", "--short"])`
- `scripts/run-code-review-agent.ts:82-86` compares the before/after status through `diffUnexpectedStatus`
- `src/shared/code-review-agent.ts:46-52` implements:
  - build a `Set(before)`
  - return only `after` lines not already present
- I validated the helper behavior directly:

  Command run:
  - `node --input-type=module -e "import { diffUnexpectedStatus } from './src/shared/code-review-agent.ts'; console.log(JSON.stringify(diffUnexpectedStatus(['M README.md'], ['M README.md'])))"`

  Result:
  - `[]`

That result is expected from the code and demonstrates the blind spot: a second mutation to an already-dirty file is invisible.

#### Proposition

Track content deltas, not just status-line appearance.

Specific fix direction:
- Compare `git diff --name-only` / `git diff --raw` before and after for tracked files.
- Separately compare untracked-file inventories.
- For maximal safety, hash file contents for all paths that are already dirty before the run and fail if their hashes change.
- If a dirty tree is unsupported, fail fast up front unless the user explicitly opts into a “dirty-tree review” mode.

#### Tests to add or update

Add helper or integration coverage for:
- dirty tracked file mutated again during review
- pre-existing untracked file modified during review
- staged file changed again during review
- deletion/rename transitions, not just `M` and `??`

Current tests only cover the clean-tree “new status entry appears” case:
- `tests/code-review-agent.test.ts:80-101`

#### Docs/skills/workflows to update

- `doc/code-review-agent.md` safety rules section
- `README.md` automated review section
- `code_reviews/README.md` notes section

### 4. Watch mode can repeatedly re-run the same expensive failing review forever

- Severity: medium
- Confidence: high

#### Why it matters

The new hard-fail behavior is sensible, but in watch mode it creates an operational loop: if a commit consistently fails review generation, the watcher will keep invoking Hermes on the same HEAD every poll interval until something changes.

That is wasteful, noisy, and brittle for a long-lived local automation loop. In the failure mode this commit is specifically trying to tighten, the result becomes a perpetual retry storm.

#### Evidence

- `scripts/run-code-review-watch.ts:17-33`
  - infinite loop
  - `if (!hasReview) runReview()`
- `scripts/run-code-review-watch.ts:47-56`
  - failed review only logs:
    - `"[crazyShell-reviewer] review run failed; will retry on next poll"`
- `code_reviews/README.md:50-52`
  - watcher triggers whenever current HEAD has no review report
- After this change, a failed run produces no report artifact, so `hasReviewForCommit` stays false and the same commit is retried indefinitely.

#### Proposition

Add commit-level failure memory and backoff.

Specific fix direction:
- Record the last failed commit and suppress repeated retries until HEAD changes, or
- implement exponential backoff / retry budget per commit, and
- optionally write a non-authoritative failure marker under `code_reviews/` or a separate state file so operators can see why review generation is stalled.

#### Tests to add or update

- Watch-mode test where `runReview()` fails twice on the same commit:
  - expected: one retry budget / backoff behavior
  - not expected: unconditional every-poll rerun forever
- Test the transition when HEAD changes after a failure

#### Docs/skills/workflows to update

- `doc/code-review-agent.md` watch-loop behavior
- `code_reviews/README.md` notes on retry semantics
