## Highest Value Findings

### 1) `.new-allowed` is keyed by canonical shot key, but the gate checks a display id

- Severity: high
- Confidence: high

#### Why it matters

This breaks the newly introduced pressure-release valve for legitimate `new` shots.

The docs, README, types, and allowlist file all describe the allowlist format as a canonical key of the form `<suite>::<slug>`. The gate, however, checks `shot.id`, which is currently a display-oriented `suite-slug` string. Result: a correctly allowlisted shot still fails `--gate`.

That is not a cosmetic mismatch. It means maintainers can follow the documented workflow exactly and still get a red CI gate.

#### Evidence

- Canonical key contract:
  - `tests-e2e-baselines/.new-allowed:6` says key format is `<suite>::<spec>-<test-slug>-<step>`
  - `src/design-report/types.ts:105-107` defines `shotKey()` as `${suite}::${slug}`
- Report builder computes the canonical key, but does not store/use it for gating:
  - `scripts/build-design-report.ts:123-125` computes `const key = shotKey(suite, slug);`
  - `scripts/build-design-report.ts:163` and `:207` set `id: \`${suite}-${slug}\``
- Gate compares against `shot.id`, not the canonical key:
  - `src/design-report/gate.ts:34-35`
- Existing allowlist test covers only the synthetic `id` path, not the documented canonical format:
  - `tests/design-report/gate.test.ts:62-66`
- Live validation:
  - `bun -e '...evaluateGate({shots:[{id:"native-design-review-component-rename-workspace-dialog-dialog-rename-workspace", status:"new", ...}], allowedNew:new Set(["native::design-review-component-rename-workspace-dialog-dialog-rename-workspace"])})'`
  - output: `{"failed":true,"failingStatuses":["new"]}`

#### Proposition

Make canonical key a first-class field on `ReportShot` and gate on that, not on `id`.

Concretely:
- add `key: string` to `ReportShot`;
- populate it from `shotKey(suite, slug)` in `scripts/build-design-report.ts`;
- keep `id` only for UI/rendering if needed;
- change `evaluateGate()` to check `allowedNew.has(shot.key)`.

This aligns the gate with the documented contract-first key format and avoids cross-module drift.

#### Tests to add or update

- Update `tests/design-report/gate.test.ts` to use real canonical keys (`web::...`, `native::...`) rather than arbitrary ids.
- Add an integration-style unit test around report-building + gate evaluation proving that a shot allowlisted in `.new-allowed` passes.
- Add a regression test specifically mirroring the current real entry in `tests-e2e-baselines/.new-allowed`.

#### Docs/skills/workflows to update

- `doc/design-report.md`
- `tests-e2e-baselines/README.md`

Both should explicitly state that the gate keys off the canonical `key`, not any UI/display identifier.

### 2) Corrupt PNGs are silently downgraded into `new` or `baseline-only`

- Severity: high
- Confidence: high

#### Why it matters

The new report explicitly distinguishes `corrupt` from `new` so postmortems are not misleading. The current implementation does not preserve that distinction.

A corrupt baseline currently becomes `new`.
A corrupt current shot currently becomes `baseline-only`.

That is operationally dangerous:
- a corrupted baseline can be mistaken for a legitimate newly added shot;
- a corrupted current artifact can look like test coverage disappeared;
- `.new-allowed` could incorrectly suppress what is actually artifact corruption.

#### Evidence

- The intended contract is explicit:
  - `doc/design-report.md:47-49` says `corrupt` means “the PNG didn't decode” and is distinct from `new`
- Current decode path collapses corruption into `null` before classification:
  - `scripts/build-design-report.ts:188-189`:
    - `const currentPng = tryDecodePng(currentBuf);`
    - `const baselinePng = baselineBuf ? tryDecodePng(baselineBuf) : null;`
  - `scripts/build-design-report.ts:191-196` passes those `null`s into `classifyShot()`
- `classifyShot()` then interprets `null` as absence, not corruption:
  - `src/design-report/shot-classify.ts:52-69`
  - only the pixelmatch catch path returns `corrupt` (`src/design-report/shot-classify.ts:105-114`)
- Existing tests validate decode failure only at the helper level, not at the classification/report integration boundary:
  - `tests/design-report/shot-classify.test.ts:120-132`
- Live validation:
  - `bun -e '...console.log(JSON.stringify({badCurrent: classifyShot({current: bad, baseline: good,...}).status, badBaseline: classifyShot({current: good, baseline: bad,...}).status}))'`
  - output: `{"badCurrent":"baseline-only","badBaseline":"new"}`

#### Proposition

Do not represent “decode failed” as plain `null`.

Use a discriminated result from decode, e.g.:
- `{ kind: "ok", png }`
- `{ kind: "missing" }`
- `{ kind: "corrupt" }`

Then map corruption to `status: "corrupt"` before absence-based classification runs.

At minimum:
- `tryDecodePng()` should not erase the difference between “buffer absent” and “buffer present but invalid”;
- `scripts/build-design-report.ts` should short-circuit to `corrupt` when a present file fails decode.

#### Tests to add or update

- Add a report-builder integration test for:
  - corrupt current + valid baseline => `corrupt`
  - valid current + corrupt baseline => `corrupt`
- Extend `tests/design-report/shot-classify.test.ts` or add a `build-design-report` focused test module to cover decode-failure propagation, not just raw helper return values.
- Add a gate test asserting `corrupt` fails gated mode.

#### Docs/skills/workflows to update

- `doc/design-report.md`

Once fixed, the docs remain valid; until fixed, they are overstating the behavior. Update only if the implementation contract changes.

### 3) `HT_TEST_BLOOM=on` / `test:native:bloom-on` no longer turns bloom on

- Severity: medium
- Confidence: high

#### Why it matters

The repo now advertises a dedicated path for running native tests with bloom re-enabled, but after the global default changed to `terminalBloom: false`, that path is effectively broken.

This is a real workflow regression:
- maintainers cannot actually exercise the bloom layer through the documented test command;
- the fixture comments and docs say one thing, the seeded settings do another;
- test determinism logic is now entangled with a product-default flip.

#### Evidence

- Native test command claims to opt bloom back in:
  - `package.json:22` — `"test:native:bloom-on": "HT_TEST_BLOOM=on ..."`
- Docs say the same:
  - `doc/design-report.md:125-129`
- Product default changed globally:
  - `src/shared/settings.ts:478-479`
    - `terminalBloom: false`
    - `bloomIntensity: 1.0`
- Fixture logic only zeroes intensity when bloom is off; it never sets `terminalBloom: true` when bloom is on:
  - `tests-e2e-native/fixtures.ts:270-273`
- So with `HT_TEST_BLOOM=on`, the override object is empty and the app still boots with `terminalBloom: false`.
- Live checks:
  - `bun -e 'import { DEFAULT_SETTINGS } from "./src/shared/settings"; console.log(JSON.stringify({terminalBloom: DEFAULT_SETTINGS.terminalBloom, bloomIntensity: DEFAULT_SETTINGS.bloomIntensity}));'`
    - output: `{"terminalBloom":false,"bloomIntensity":1}`
  - fixture-equivalent snippet with `HT_TEST_BLOOM=on` produced `{}` overrides

#### Proposition

In the native fixture:
- when `HT_TEST_BLOOM=on`, explicitly seed `terminalBloom: true`;
- if no explicit intensity override is provided, preserve or re-seed a non-zero `bloomIntensity`.

Separately, I would recommend keeping the test harness concern isolated in the fixture rather than using test flakiness as justification for a shipping-default change, unless product/UX also explicitly wants bloom off by default.

#### Tests to add or update

- Add a focused test around the settings-seeding logic in `tests-e2e-native/fixtures.ts`:
  - default => bloom off
  - `HT_TEST_BLOOM=on` => `terminalBloom: true`
- Add a native smoke/assertion that the app reports bloom enabled under `test:native:bloom-on`.
- Add a regression test for env-merging precedence when `HT_TEST_SETTINGS_JSON` and `HT_TEST_BLOOM` are both set.

#### Docs/skills/workflows to update

- `doc/design-report.md`
- any repo workflow note that references `test:native:bloom-on`

The docs currently promise behavior that the fixture does not deliver.

### 4) `manifest.json` claims to preserve source `test` and `step`, but promotion fabricates them heuristically

- Severity: medium
- Confidence: high

#### Why it matters

This is a contract-quality problem more than an immediate crash bug, but it matters because the new manifest is being positioned as an authoritative artifact.

The schema and docs say manifest entries contain source `test` and `step`. Promotion does not preserve those values from source metadata; it reverse-parses filenames and writes the same lossy suffix into both fields. That weakens the contract, makes audits/debugging less trustworthy, and violates the “typed contracts over ad hoc payloads” guidance in the architecture standard.

If future tooling starts relying on exact `test` / `step` fields, it will be built on false data.

#### Evidence

- Manifest schema claims semantic source fields:
  - `src/design-report/types.ts:96-99`
- Docs reinforce that contract:
  - `doc/design-report.md:61-63`
- Promotion explicitly does a best-effort reverse parse from filename:
  - `scripts/promote-design-baseline.ts:80-89`
- The committed manifest shows the result is already lossy/fabricated:
  - `tests-e2e-baselines/manifest.json:5-9`
    - `"slug": "demos-demo-3d-demo-3d"`
    - `"test": "demo-3d-demo-3d"`
    - `"step": "demo-3d-demo-3d"`

Those are not the original source test title and step; they are a filename-derived duplicate.

#### Proposition

Generate the manifest from structured report/index metadata, not from reverse-parsed PNG filenames.

Best fix direction:
- persist the structured `spec` / `test` / `testSlug` / `step` alongside promoted shots during `baseline:design`;
- build `manifest.json` from that structured source;
- fail promotion if the structured metadata needed for the manifest is unavailable.

If you intentionally only want filename-derived data, then narrow the manifest contract and rename those fields so they do not imply exact source provenance.

#### Tests to add or update

- Add a promotion test that starts from sample structured shot metadata and asserts exact manifest `test` / `step` round-trip.
- Add a regression case where both test title and step contain dashes, proving they remain distinct after promotion.
- Expand `tests/design-report/manifest-and-allowed.test.ts` beyond pure I/O into contract-level generation coverage.

#### Docs/skills/workflows to update

- `doc/design-report.md`
- `tests-e2e-baselines/README.md`

Either the docs need to be narrowed, or the code needs to fulfill the stronger manifest contract.
