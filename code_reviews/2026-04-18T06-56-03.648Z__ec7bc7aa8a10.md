# crazyShell Reviewer Report

## Executive Summary

This first-pass review focused on the sideband/panel pipeline and the web-mirror client, because the recent commit stream is concentrated there and these paths are easiest to regress without immediately breaking the PTY happy path.

The highest-value concerns are all in parity and lifecycle handling between the native webview and the web mirror:

- the web mirror’s panel DOM reconciliation is materially less complete than the native `Panel`/`PanelManager` path;
- pane teardown in the web mirror leaks `ResizeObserver` instances and pending timers;
- `canvas2d` rendering on both native and web-mirror sides can apply frames out of order and never closes decoded bitmaps;
- web-mirror panels still lack the documented close affordance, so remote behavior diverges from native behavior and from the documented sideband contract.

The codebase is directionally strong: the sideband parser is bounded and defensive, the recent fixes around update-after-close and binary/meta races are good, and the reducer/store split in the web mirror is a meaningful architectural improvement. The remaining problems are mostly “two implementations drifting apart” issues rather than foundational design failures.

## Highest Value Findings

### 1. Web-mirror panel reconciliation is still semantically weaker than the native panel system

- Severity: high
- Confidence: high

#### Why it matters

The repository rules and docs are explicit that panels are overlays with defined behavior, and the web mirror is supposed to mirror live panel state rather than invent its own reduced semantics. Right now the native path and web-mirror path do not apply the same panel contract.

That creates drift in exactly the areas that users notice first: interactivity toggles, border radius, z-order, drag/resize affordances, and re-creation semantics when a panel is replaced with the same ID. This is the kind of split-brain UI behavior that is hard to diagnose because the PTY remains correct while the overlay layer behaves inconsistently across clients.

#### Evidence

Native path handles a materially richer mutable panel surface:

- `src/views/terminal/panel.ts:97-116`
  - `updateMeta()` applies `x`, `y`, `width`, `height`, `zIndex`, `opacity`, `borderRadius`, `interactive`, and inline `data`
- `src/views/terminal/panel.ts:189-206`
  - `applyMeta()` re-applies `zIndex`, `borderRadius`, `draggable`, `resizable`
- `src/views/terminal/panel-manager.ts:72-85`
  - a non-`update` message with an existing ID removes and recreates the panel

Web-mirror path only applies a narrow subset when DOM already exists:

- `src/web-client/main.ts:575-586`
  - existing panel path updates only `x`, `y`, `width`, `height`, `opacity`, and raw `data`, then returns
- Missing there:
  - no `zIndex`
  - no `borderRadius`
  - no rebind/unbind for `interactive`
  - no reconfiguration for `draggable` / `resizable`
  - no recreation on same-ID new create with different type or chrome
- Creation path is richer, but only on first materialization:
  - `src/web-client/main.ts:591-635`

State/store already preserves the richer meta fields:

- `src/web-client/store.ts:477-540`
- `src/bun/web/state-store.ts:43-71`

The docs describe a broader contract than the web mirror currently honors:

- `doc/system-sideband-protocol.md:102-109`
- `doc/system-sideband-protocol.md:196-207`
- `doc/how-to-use-sideband.md:202-205`
- `doc/how-to-use-sideband.md:423-426`

#### Proposition

Make panel DOM reconciliation a contract-complete operation, not a best-effort style patch.

Concretely:

1. Extract a shared “apply mutable panel meta” helper for the web mirror that mirrors native `Panel.applyMeta()` behavior.
2. On existing DOM:
   - apply `zIndex` and `borderRadius`
   - enable/disable interactive handlers and CSS classes
   - enable/disable drag and resize chrome and handlers
3. If a non-`update` meta arrives for an existing ID, recreate the DOM node, matching `PanelManager.handleMeta()` semantics.
4. Treat the native webview implementation as the reference behavior and make web-mirror parity explicit in code comments/tests.

This is aligned with the architecture guide’s “typed contracts over ad hoc payloads” and “explicit module ownership” guidance. Right now the contract exists, but the two renderers do not actually implement the same one.

#### Tests to add or update

Add DOM-level tests for web-mirror panel lifecycle, not just reducer tests:

- create panel → update `zIndex` / `borderRadius` → assert DOM reflects change
- update `interactive: false -> true -> false` → assert handler presence/absence and class toggling
- update `draggable` / `resizable` → assert chrome and behavior change
- send non-`update` create with same ID but different type/chrome → assert DOM is recreated
- add negative tests that mirror native `PanelManager` semantics on unknown update / known recreate

Likely targets:
- new `tests/web-client-panels.test.ts`
- extend `tests/web-client-store.test.ts` only for state semantics, not DOM behavior

#### Docs/skills/workflows to update

If fixed, update:

- `doc/how-to-use-sideband.md`
- `doc/http-web-ui-refactor.md`
- `doc/system-canvas-panels.md`

Specifically document that web-mirror panel behavior is intentionally kept in parity with native `PanelManager`, including same-ID recreate behavior and mutable update fields.

### 2. Web-mirror panes leak `ResizeObserver` instances and deferred resize timers on teardown

- Severity: high
- Confidence: high

#### Why it matters

This is a lifecycle bug in a long-lived UI process. The repo rules explicitly call out “cleanup/finalizer logic must never throw” and “avoid leaked listeners, timers, sockets.” Pane churn is a core interaction in HyperTerm Canvas, so any leak attached to pane create/dispose will compound under normal usage.

Detached observers retaining pane DOM and closure state are exactly the kind of invisible cost that turns into rising memory use and stale behavior over long sessions.

#### Evidence

A `ResizeObserver` and a timer are created per pane:

- `src/web-client/main.ts:327-352`
  - `const ro = new ResizeObserver(...)`
  - inner `resizeTimer = setTimeout(...)`
  - `ro.observe(termEl)`

But neither the observer nor the timer is retained for cleanup:

- `TermRef` in `src/web-client/main.ts:132-139` has no observer/timer fields
- `disposePane()` in `src/web-client/main.ts:361-370` only:
  - disposes the terminal
  - removes the element
  - deletes the map entry

There is no corresponding:

- `ro.disconnect()`
- `clearTimeout(resizeTimer)`

#### Proposition

Treat observer/timer ownership as part of pane lifetime.

Concretely:

1. Extend `TermRef` to own:
   - `resizeObserver`
   - `resizeTimer`
2. In `disposePane()`:
   - `disconnect()` the observer
   - clear any pending timer
   - then dispose xterm and remove the DOM
3. On connection reset / workspace switch paths, verify pane disposal is the single owner of cleanup.
4. Add a small invariant comment: “all non-DOM async resources created in `createPane` must be released in `disposePane`.”

#### Tests to add or update

Add a web-mirror pane lifecycle test that:

- creates a pane
- disposes it
- verifies no further `surfaceResizeRequest` messages are sent after removal when geometry changes or observer callback is stimulated

If happy-dom makes `ResizeObserver` awkward, a mocked observer shim is sufficient.

Likely target:
- new `tests/web-client-main.test.ts` or `tests/web-client-pane-lifecycle.test.ts`

#### Docs/skills/workflows to update

Update:

- `doc/http-web-ui-refactor.md`

Specifically call out observer cleanup as part of pane lifecycle ownership, because that document already frames web-mirror correctness in terms of reducer/view separation and reconciling views.

### 3. `canvas2d` streaming can render frames out of order and never closes decoded bitmaps

- Severity: medium
- Confidence: high

#### Why it matters

This is both a correctness and performance issue in a performance-sensitive repository.

For a live stream, `createImageBitmap(blob).then(...)` is asynchronous. Without a generation check, an older frame decode can complete after a newer frame and overwrite the canvas with stale content. Separately, never calling `bitmap.close()` leaves cleanup to GC rather than explicit resource finalization.

This affects both renderers, so native and web mirror can both exhibit stale-frame flashes or accumulating decode resources under sustained streaming.

#### Evidence

Web mirror:

- `src/web-client/panel-renderers.ts:122-129`
  - `createImageBitmap(blob).then((bitmap) => { ... drawImage(bitmap, 0, 0); })`
  - no generation/version guard
  - no `bitmap.close()`

Native webview:

- `src/views/terminal/content-renderers.ts:172-179`
  - same pattern
  - no generation/version guard
  - no `bitmap.close()`

The sideband docs and demos explicitly support live visual updates:

- `doc/how-to-use-sideband.md:197`
- `scripts/demo_webcam.ts:223-295`

So this is not a theoretical edge case; the codebase is actively encouraging streamed visual updates.

#### Proposition

Make canvas rendering monotonic and explicit about cleanup.

Concretely:

1. Track a render generation per canvas/content element.
2. When a new frame is scheduled, increment the generation.
3. In the `createImageBitmap(...).then(...)` continuation:
   - bail out if generation is stale
   - draw only if still current
   - always `bitmap.close()` in `finally`
4. Consider debouncing or “latest-frame wins” semantics if decode pressure becomes visible.

This should be implemented in both:
- `src/web-client/panel-renderers.ts`
- `src/views/terminal/content-renderers.ts`

#### Tests to add or update

Add focused tests with a mocked `createImageBitmap` that resolves out of order:

- frame A starts
- frame B starts
- B resolves first, then A resolves
- assert canvas ends with frame B, not A

Also assert `bitmap.close()` is called.

Likely targets:
- new `tests/web-client-panel-renderers.test.ts`
- new `tests/content-renderers.test.ts`

#### Docs/skills/workflows to update

Update:

- `doc/how-to-use-sideband.md`

Not a behavioral API change, but worth documenting that `canvas2d` is latest-frame-wins and designed for live streaming, which clarifies expectations for demo/script authors.

### 4. The web mirror still lacks the documented panel close affordance

- Severity: medium
- Confidence: high

#### Why it matters

The docs and native implementation say draggable panels have a close button and emit a `close` event so scripts can clean up. The recent webcam fix explicitly leans on this behavior. The web mirror does not expose that affordance, so remote users cannot dismiss a panel or trigger the script’s cleanup path from the mirror.

That is a clear UX parity gap and a real operational limitation for “runaway panel” scenarios.

#### Evidence

Native implementation has explicit close UI and event emission:

- `src/views/terminal/panel.ts:43-49`
- `src/views/terminal/panel.ts:62-63`

Docs describe the close affordance as standard behavior:

- `doc/system-canvas-panels.md:87-96`
- `doc/system-sideband-protocol.md:145-148`
- `doc/how-to-use-sideband.md:423`

Web-mirror panel creation does not create any close control:

- `src/web-client/main.ts:612-629`
  - drag handle and resize handle only
- `src/web-client/panel-interaction.ts:55-226`
  - forwards mouse, drag, resize
  - no close helper / no close event source
- `src/web-client/client.css:427-483`
  - drag and resize styling only
  - no close button styling

#### Proposition

Add close-button parity to the web-mirror panel chrome.

Concretely:

1. Add a close control to draggable panel chrome in `src/web-client/main.ts`.
2. Style it in `src/web-client/client.css`.
3. On click:
   - emit `panelMouseEvent` / `close` back to the server
   - optimistically hide local DOM only if that matches existing event flow
4. Add a clear statement in code/comments about whether web-mirror close is authoritative or merely requests native close behavior.

Given the repo’s “PTY is source of truth; overlays are ephemeral” rule, the safest design is:
- emit close request/event,
- let server/native echo the resulting panel state change back,
- avoid inventing an unacknowledged local-only deletion path.

#### Tests to add or update

Add a DOM interaction test that:

- creates a draggable web-mirror panel
- clicks the close control
- asserts a `close` event is sent with the right `panelId` / `surfaceId`

Also extend interaction coverage to include “drag/resize/close still fire without `interactive: true`,” because the docs explicitly say they should.

Likely targets:
- `tests/web-client-panel-interaction.test.ts`
- new web-panel DOM test

#### Docs/skills/workflows to update

Update:

- `doc/system-canvas-panels.md`
- `doc/how-to-use-sideband.md`
- `doc/http-web-ui-refactor.md`

If the mirror is intentionally not meant to support close, that exception needs to be documented explicitly. Right now the docs imply parity.

## Verdict

- The recent sideband/web-mirror fixes improved the right failure modes, but the native and web-mirror panel implementations are still not behaviorally equivalent.
